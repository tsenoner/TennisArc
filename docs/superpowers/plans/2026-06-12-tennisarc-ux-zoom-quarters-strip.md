# TennisArc UX overhaul — unified proposal

## 1. The unified interaction model

Everything follows from one structural change: the match insight becomes a slim, non-occluding **strip above the wheel** instead of a panel/sheet. Because selecting a match no longer buries the chart, touch and desktop converge on one grammar — and the freed-up "second tap" becomes the zoom gesture the owner asked for.

- **Tap an arc → select.** Pins the player's path and opens the match strip. Single tap on touch too: the old pin-first/second-tap-opens dance existed only because the sheet covered the wheel; the strip *is* the preview.
- **Tap the same arc again → zoom.** State-based, not timing-based: if the tapped arc is already `selectedNodeId`, call `setFocus(sectionOf(id))` (node itself if it has children, else its parent). No 350 ms window, no insight flash, no scrim swallowing taps, no dead leaf taps — this dissolves every blocker the double-tap design had.
- **Tap a corner quarter label → focus that quarter.** The discoverable entry; same `data-action="focus"` branch.
- **Tap the hub while focused → zoom out one level.** Reorder the inspect branch so `id === state.focusId` is checked first — the currently shadowed reset clause finally becomes reachable.
- **Pinch (phone) → magnify.** Ephemeral geometric zoom, orthogonal to semantic focus, auto-reset on any geometry change.
- **One back ladder, one layer per press** — identical for ESC, ✕/scrim, crumbs: menu → detail tier → match strip → lens drawer → pin → pinch zoom (k>1) → focus. Browser Back / iOS back-swipe exits focus (one history entry per focus session).

```
IDLE ─tap arc─▶ SELECTED(pin+strip) ─tap same arc─▶ FOCUSED(section)
IDLE ─tap corner label──────────────────────────▶ FOCUSED(quarter)
SELECTED ─"Details ▾"─▶ DETAIL tier (in-flow / bottom sheet)
FOCUSED ─tap hub─▶ FOCUSED(parent) … ▶ IDLE
FOCUSED ─crumb · Back-swipe · ESC─▶ IDLE
any ─pinch─▶ MAGNIFIED(k≤4), auto-reset on focus/slam change
country lens: arc tap still toggles nation; labels/crumbs still zoom
```

## 2. Quarter labels

**Recommended: corner-anchored straight SVG handles** in the dead circle-in-square wedges (disc r=342 in the 700 viewBox; quarter mid-angles point exactly at the corners). Per quarter, inside the `translate(350,350)` group: `<g class="q-owner" data-action="focus" data-id="r.0.0" data-occupant="{pid}">` with a transparent hit rect (~150×56, **invariant: never inside r=342** — comment + render test), a 10×7.5 `<image>` flag (never emoji), `.q-name` surname 14px/700 with the `.ring-label` halo (stroke `var(--bg)` 2.6px), `.q-cap` "Q1 · seed 1" 8.5px `var(--dim)`. TR=`r.0.0`, BR=`r.0.1`, BL=`r.1.0`, TL=`r.1.1`; right corners `text-anchor=end` at x=336, left mirrored.

**Owner rule:** drawn top seed — min non-null seed over the quarter's 16 leaf occupants, tie → better ranking, no seeds → best ranking, all-TBD → caption-only "Q1". Matches the owner's RG26 reading exactly (Sinner/FAA/Djokovic stay despite elimination). Eliminated owner: `.q-out` dims the name; **no "· out" text** (state lives in color + `aria-label`).

**Behavior:** hover lights the owner's path free via the existing `[data-occupant]` pointermove — add `.sunburst.arc-dim-mode .q-owner:hover { opacity: 1 }` so the hovered label never self-dims. Tap focuses the quarter (labels are not arcs, so this works on every lens). Hidden entirely when `state.focusId` is set. Phone (≤720px): name-only at 17 SVG units (~8.9px effective); hide caption + flag (they'd render ~5.5px).

Kept on the table: **A — curved ribbons** outside the ring (strongest circular language, but costs ~12% disc area and curved 11px text loses on legibility); **C — tinted quarter band** (the brace IS the quarter, but breaks the outermost-ring-=-R128 grammar and the round-axis).

## 3. Focus & zoom

**Grammar:** as in §1 — re-tap selected arc, corner labels, hub-tap-out, plus the strip's always-visible `⊕ Zoom` button (accented, replacing the buried ghost `.mi-focus`). Focus to any non-leaf depth is allowed; quarters are the primary unit.

**Full-area fix (3 lines, layout.ts:33-39):** add `const ky = fy0 > 0 && fy0 < radius - 1e-9 ? radius / (radius - fy0) : 1;` and change to `y0 = Math.max(0, (n.y0 - fy0) * ky)`, same for y1. A focused quarter of a 128 draw goes from 6 rings × 42.75px capped at 0.75R to 6 × 57px filling r=342 exactly. Round axis and label fitting recompute from arc geometry automatically; many labels flip radial→curved (thicker rings), so budget the screenshot pass. Restore the center pill in focus mode (focused occupant; section title fallback), add `anchors.delete(state.focusId)` beside the existing root delete, and recompute `floatIdle` as `!hover && !pinned` so the player isn't shown twice.

**Back affordances:** (1) **Crumb row** — slim in-flow chip line at the top of `.sunburst` (same slot pattern as the strip; never overlaps the 12-o'clock axis tab or `.ro-float`): `‹ Full draw` + ancestor chips + current section name from `sectionTitle()` ("Top half", "Sinner's quarter" via the §2 derivation, "QF section"). Relax the focus-branch guard to `el.dataset.id !== undefined` so the empty-id "Full draw" chip works. (2) **Hub tap** zooms out one level. (3) **ESC** unchanged in position. (4) **History, V1-simple:** push exactly ONE entry on focus entry (`#r.0.0`), `replaceState` on level change, `history.back()` on clear only if we pushed; popstate routes through `setFocus`+`draw()`; slam/tour switch scrubs the hash. Back-swipe now exits focus instead of the app; multi-level clears stay one-shot. `setFocus` normalizes `"r"`/unresolvable ids to undefined.

**Transition stance:** ship snap (matches the zero-animation idiom). Optional follow-up commit: one-shot ghost cross-fade (clone old svg, scale toward focused centroid computed via `getScreenCTM` — never % transform-origin), behind `prefers-reduced-motion`, fully revertable.

## 4. Mobile zoom

**Recommended: two-finger pinch/pan (design B) as an ephemeral magnifier, with semantic focus remaining primary.** Focus fixes structure and hit targets, but font caps + the ~0.56× phone render keep even focused labels at 7–10 CSS px — only geometric magnification makes names readable, so the ask is real, not redundant with focus. Grammar is collision-free: one finger keeps drag-to-explore, two fingers are currently swallowed, the tap verbs are untouched.

Mechanics: `src/zoom.ts` pure math (`pinchUpdate`/`panUpdate`, k clamped [1,4], pan clamped `x,y ∈ [700(1−k), 0]`, snap below 1.05), `<g class="zoom-layer">` wrapping the translate group, transform written as an **SVG attribute** (vector-crisp; never CSS transform — WebKit rasterizes). View lives module-level in `createApp()`, never in AppState/URL; reset whenever the geometry key (slam|tour|focusId) changes, **and inside the hub-reset branch** so the nuclear reset is honest. Critique fixes baked in: gate pointers on `closest(".sunburst")` (not the svg — the docked readout strip eats second fingers), set the click-suppression flag at second pointerdown (not only on move), defer `draw()` while two pointers are down (flush on gesture end), `.zoom-reset` chip gets `display:inline-flex` in the ≤720px block at `bottom: calc(52px + safe-area)`, z-36, and a zoom-reset rung joins ESC between pin and focus. **Gate on on-device profiling**; fallback is hiding `.arc-label/.ring-label` during the gesture, never CSS-transform blur.

# Appendix: per-area designs and adversarial critiques (workflow output)


## quarter-labels

**Recommendation:** Alternative B — corner-anchored straight SVG labels acting as tappable quarter handles, with owner = drawn top seed.

**Rationale:** B is the only option that costs zero wheel radius: the corner wedges are geometrically guaranteed dead space (disc touches the square only at the cardinal points), and the quarter mid-angles point exactly at them, so the placement is principled, not decorative. Straight horizontal text at 14px beats every curved option on the owner's top criterion (legibility), supports <image> flags without the WebKit textPath-emoji trap, and reuses the .ring-label halo/token recipe verbatim for cross-theme consistency. Because the labels live inside the viewBox they scale with the chart, avoiding the HTML-overlay misalignment bug class already paid for once. Interaction comes nearly free: data-occupant plugs into the existing pointermove highlight/readout, and data-action="focus" reuses the existing app.ts:368 branch — making the labels the discoverable entry point to quarter zoom that the hidden '⊕ Focus this section' button never was, which is the keystone of the companion zoom redesign. Owner semantics 'drawn top seed' is what the user's RG26 example demands (Sinner and Auger-Aliassime are eliminated yet expected): it matches how the tennis world names quarters ('Sinner's quarter' survives his exit), is stable for the whole tournament, and the .q-out dimmed state still tells the upset story honestly.

### Alternative: A — Curved textPath ribbons just outside the outer ring (45/135/225/315°)

Shrink the layout radius from SIZE/2-8=342 to SIZE/2-30=320 (app.ts:156) and draw four curved textPath labels at r≈332, each centred on a quarter mid-angle (π/4, 3π/4, 5π/4, 7π/4), spanning ~0.5 rad. Reuse the existing curved() idiom from render.ts:103-112 including the bottom-half flip (revT, render.ts:101). Text: "1 · SINNER" (seed + surname, uppercase, 11px, letter-spacing .04em), class .q-arc-label with the ring-label halo recipe (fill var(--dim) or var(--text), paint-order:stroke, stroke var(--bg) 2.6px). No flags (WebKit won't paint emoji on textPath; a rotated <image> flag beside curved text is fiddly). Each label's <path> def gets a sibling transparent hit arc (a 22px-thick annulus segment, data-action="focus" data-id="r.0.0" data-occupant={ownerId}) so tapping the ribbon focuses the quarter. Hidden when state.focusId is set.

**Pros:** Strongest circular language — the label literally arcs over its quarter, association is unambiguous; Reuses existing curved-label machinery and halo standards nearly verbatim; Hit annulus gives a continuous 90° tap target for tap-to-focus

**Cons:** Shrinks the wheel ~6.4% in radius (~12% area) on every viewport — worst on phones where the wheel is already small and the owner has fought for ring legibility; Bottom-half labels flip reading direction (existing convention, but these are the four most prominent words on the chart); No flag possible without a separate rotated <image>, and seed+name at 11px curved is less legible than straight text — legibility is the owner's top criterion; Adds a fifth label vocabulary (curved meta-label outside the data rings) competing with the 12-o'clock round axis

### Alternative: C — Outer quarter band: four 90° arc segments forming a thin grouping annulus

Shrink layout radius to 318 and add a dedicated 16px annulus (r 326-342) of four arc segments (one per quarter, 2px angular gaps), each tinted with a barely-there per-quarter wash (8% alpha of the theme arc palette) and carrying a curved textPath "SINNER · Q1" at its mid-radius, 10px, ring-label halo. ### Critique verdict

The recommended design (B — corner-anchored quarter handles) fundamentally holds: every load-bearing claim checks out against the real code (focus branch wiring, orientation math, corner clearance, data semantics, test compatibility), and it matches the owner's idioms (string-rendered SVG, data-action delegation, ring-label halo recipe, no textPath emoji). The RG26 data confirms the "drawn top seed" semantics exactly (Sinner/FAA/Djokovic out, Zverev champion). However, shipping it as written creates a mobile trap: the labels make one-tap focus prominent while focus exit is verifiably ESC-only today (no reachable reset path on touch), so an exit affordance must land in the same change. Two CSS-level polish bugs (hover self-dim, illegible phone caption/flag) would also fail the owner's screenshot review and need fixing before it ships.

- **[blocker]** Focus exit is unreachable on touch, and design B hands every phone user a one-tap entry into that dead end. Verified: every arc carries data-action="inspect" with a non-empty data-match (render.ts:163-164), and the inspect branch (app.ts:343) precedes the reset clauses in the else-if chain, so `id === 'r'` / `id === state.focusId` (app.ts:375) never fire from arcs; the <g data-action="reset"> (render.ts:171) is never the resolved target because all its painted children either carry their own data-action or are pointer-events:none (app.css:147,152,76); empty-svg taps hit the `!el` branch which clears only pinnedId (app.ts:284-289). The sole working exit is window-level ESC (app.ts:427) — which does not exist on iOS. The design lists this as a risk to fix 'in quick succession'; that is too weak.

### Critique verdict

The recommended design (A) fundamentally holds: the layout.ts radial-rescale fix is correct and minimal, the reset-clause shadowing it builds on is real, the data/orientation assumptions are exactly right (verified against RG26 JSON), and the breadcrumb/history/gesture stack fits the owner's taste and the codebase idioms. However, the history-ownership model is structurally broken for multi-level focus (a single history.back() cannot clear a depth-2 focus, so ESC, the hub reset, and the "Full draw" crumb all under-deliver), and the as-specced "Full draw" crumb is dead under the existing focus-branch guard. The double-tap reinterpretation also has two unexamined touch collisions: the full-screen mi-scrim swallows the second tap whenever tap-1 opened the sheet, and a fast tap-tap (the established touch grammar, codified in app.test.ts:149-161) now zooms or no-ops instead of opening the match insight. All fixable without changing the architecture.

- **[blocker]** setFocus's history model breaks for multi-level focus. The spec says entering/deepening focus pushes one entry per level, and clearing calls history.back() once when history.state?.f exists. At focus depth 2 ('r.0' then 'r.0.0'), the 'Full draw' crumb, the hub reset (app.ts:375-376 semantics: clears focus+match+pin together), and the ESC focus arm (app.ts:427, currently clears focusId entirely in one press) would all pop only ONE level — the popstate handler then re-sets focusId to the intermediate id, so 'clear' silently becomes 'zoom out one'. ### Alternative: A. Semantic-only: focus IS the zoom (no geometric transform)

Ship no pinch code. Phone zoom = problem 2's tap-to-focus on quarters/sections, made sufficient by three changes: (1) layout.ts radial rescale `y' = (y - fy0) * R / (R - fy0)` so a focused quarter fills the whole disc (rings grow from 42.75 to 57 viewBox units for a quarter); (2) raise focus-mode font caps in render.ts fitArcLabel paths (curved cap 13→17px, radial cap 11→14px when state.focusId is set — pass a `focused` flag into the label fitter) so the extra ring thickness is actually spent on legibility; (3) tap forgiveness for the tiny R128 ring: in app.ts's click handler, when the click target is the bare <svg>/<g> (no [data-action]), convert the click to viewBox coords via `svg.getScreenCTM().inverse()`, compute (r, θ) around (350,350), and look up the containing LayoutArc from the arcs array already in draw()'s closure (expose it via a module-level `lastArcs` ref); if r is within 12 units of the rim or inside a padAngle gap, dispatch the inspect branch for the nearest arc. No new CSS, no new state fields.

**Pros:** Zero gesture code, zero transform/raster risk on WebKit; nothing competes with the existing one-finger pointermove explore or the two-tap pin-then-open grammar; Fixes the real hit-target problem: a quarter focused at full disc gives R128 leaves ~37px-wide chords on a 393px phone (vs ~9px today); Smallest diff; entirely screenshot-verifiable (focused quarter before/after, label sizes)

**Cons:** Does not deliver what the user literally asked for; labels stay capped at ~9-10 CSS px on a phone (700 viewBox → ~390px render, 0.56x) even when focused — fine for tap targets, marginal for reading; No way to lean into one match's neighborhood without changing app state (focus is a layout change, not a look); Tap-forgiveness polar lookup must special-case focus mode and the country lens

### Alternative: B. Two-finger pinch/pan on a persistent SVG zoom layer, ephemeral view state, reset-on-focus (semantic stays primary)

Strict grammar: ONE finger = explore (pointermove readout/highlight, unchanged); TWO fingers = pinch-zoom and pan; DOUBLE-TAP = nothing new (it already means pin-then-open insight via the 800ms lastTouchTs heuristic — do not collide). Geometric zoom is a transient magnifier, never app state: module-level `view = {k:1, x:0, y:0}` in app.ts, NOT in AppState, never in a future URL. render.ts wraps the chart in `<g class="zoom-layer">` around the existing `<g transform="translate(350,350)" data-action="reset">`; the gesture writes `transform="translate(x,y) scale(k)"` as an SVG ATTRIBUTE (vector re-render, crisp text at any k — never CSS transform, which WebKit rasterizes and blurs). k clamped [1,4], snap to identity below 1.05; pan clamped so the scaled 700-square always covers the viewBox: x,y ∈ [700(1−k), 0]. Pointer math in viewBox units via getScreenCTM().inverse() (avoids the fixed-px-vs-scaled-SVG bug class). While 2 pointers are down, the explore pointermove handler early-returns and a 250ms post-gesture guard suppresses the synthetic click so a pinch never pins/inspects. When k>1, a small reset chip appears bottom-left of .chart — `<button class="zoom-reset" data-action="zoom-reset" hidden>⤾ reset zoom</button>`, styled like panel-fab but ghost (1px var(--line) border, var(--panel) bg, 13px), z-index 36 (below panel-fab 37), bottom: calc(14px + env(safe-area-inset-bottom)). Interplay rule: any GEOMETRY-changing state resets view to identity (focusId set/cleared, slam/year/tour switch, data-action reset); selection state (pin, inspect, lens, country) keeps the zoom. After every draw(), re-apply the transform and chip visibility as a post-render fixup next to the pin re-light (app.ts:225-228). touch-action:none on the svg already routes both fingers to pointer events; keep viewport user-scalable on (accessibility escape outside the chart).

**Pros:** Continuous magnification is the only thing that makes outer-ring names actually readable on a phone (13 viewBox px ≈ 7 CSS px → 14-28px at k=2-4); semantic focus cannot exceed the font caps; Pinch is the universal mobile idiom — zero discoverability cost, no new chrome except one reset chip; Coherent split: focus answers 'show me this quarter', pinch answers 'let me read this'; the reset-on-focus rule prevents stale-transform confusion; SVG-attribute transform keeps the owner's legibility bar: text re-rasterizes crisp at every scale

**Cons:** ~120-150 lines of gesture + clamp math and several edge cases (pointercancel, mid-gesture draw() from a resolving fetch, click suppression); Per-frame vector repaint of ~250 paths + ~127 halo labels during the gesture — needs rAF throttling; worst-case jank on older iPhones; Playwright WebKit cannot synthesize a real pinch; gesture verification needs dispatched PointerEvents or unit-tested pure math plus a manual device check

### Alternative: C. Compositor pinch (CSS transform) with crisp bake-on-settle

Same grammar and state as B, but during the gesture apply `transform: translate(...) scale(k)` as a CSS style on the .chart > svg (compositor-only, guaranteed 60fps), accepting WebKit's rasterized blur while fingers move; on pointerup, 'bake' the final view into the zoom-layer's SVG transform attribute and clear the CSS transform, restoring crisp vector text. Requires converting between CSS px (gesture space) and viewBox units (bake space) — two coordinate systems with the meet letterbox offset between them.

**Pros:** Immune to repaint jank during the gesture regardless of label count; End state identical to B: crisp at rest

**Cons:** Visible text blur mid-gesture — directly against the owner's legibility taste; Two transform code paths and a CSS-px↔viewBox conversion is exactly the fixed-px vs scaled-SVG mismatch class this codebase already got burned by; More code than B for a benefit (gesture-time fps) that profiling may show is unnecessary

### Alternative: D. Stepwise zoom chips (+/−) with one-finger pan when zoomed

No multi-touch. Two 40px round buttons (`data-action="zoom-in"/"zoom-out"`, classes .zoom-btn, stacked bottom-left, z 36) step k through 1 → 1.6 → 2.5 → 4, zooming toward the disc centre (or the pinned arc's centroid when a pin exists). When k>1, one-finger drag on the svg pans instead of exploring (pointerdown+move with a 8px slop threshold; below threshold it's still a tap), and hover-explore is desktop-only. Same zoom-layer attribute transform and reset rules as B.

**Pros:** Fully discoverable, screenshot-trivial, no multi-touch math, testable in Playwright with plain clicks; Works one-handed

**Cons:** Sacrifices the signature one-finger drag-to-explore whenever zoomed — the readout/highlight system goes dead exactly when users are leaning in; Zoom-toward-centre without pan-to-target makes reaching a specific outer arc a multi-step chore; pinch does it in one motion; More permanent chrome on an already busy phone layout (chips + panel-fab + sheets)

### Critique verdict

The recommended design (B: two-finger pinch/pan via an SVG-attribute transform on a persistent zoom-layer, ephemeral module-level view state, reset-on-geometry) holds up against the real code. Every load-bearing claim checked out: the single translate group to wrap (render.ts:171), touch-action:none routing both fingers (app.css:70), the if/else click chain where inspect shadows the focused-arc reset (app.ts:343/375), the pin re-light fixup slot for applyView (app.ts:225-228), the z-band ladder leaving 36 free, the layout.ts radii-shift-only underfill premise, and no existing test asserting svg group structure. The issues found are implementation-level — a pinch-gating selector that misses the docked readout strip, a click-suppression hole for low-movement pinches, chip CSS/placement bugs, and a reset-grammar inconsistency — none invalidates the architecture, and the acknowledged perf risk is the one thing that genuinely needs on-device profiling before commitment.

- **[major]** Pinch gating on closest(".sunburst svg") misses the second finger when it lands on the docked readout strip: at ≤960px the float readout renders as an in-flow strip INSIDE .sunburst above .chart (app.ts:215-216, app.css:196-200) with pointer-events:none (app.css:165), so that pointerdown targets the .sunburst div, never enters the pointers Map, and the pinch silently fails for top-of-chart pinches — a very natural finger placement on a phone where the chart fills most of the viewport.
  - Fix: Gate Map membership on closest(".sunburst") (or .chart) instead of the svg, while keeping all coordinate math in viewBox units via the svg's getScreenCTM(). Browser page-zoom is still blocked because the other finger sits on the touch-action:none svg.
- **[major]** Click-suppression hole: lastGestureTs is only written inside 2-pointer pointermove. A two-finger touch with sub-threshold or coalesced movement ends without ever setting it, so a synthesized click on the first finger's arc slips past the 250ms guard and pins/inspects (app.ts:343-365) — exactly the accidental-tap class the design says it prevents. The design's own risks section half-acknowledges this but the implementation spec doesn't fix it.
  - Fix: Set lastGestureTs (or a sticky pinched flag cleared on next pointerdown) the moment the second pointerdown lands and again on each pointerup while 2 pointers are tracked — not only on pointermove.
- **[major]** Per-frame repaint cost is real and unmeasured: an attribute transform on the inner <g> invalidates ~250 paths plus ~127 paint-order-stroke halo labels (app.css:146-152) per frame; stroked text is among WebKit's most expensive paints. The design correctly defers alternative C, but ships rAF throttling as the only mitigation with no profiling gate, on a feature whose entire value is smooth direct manipulation.
  - Fix: Make on-device profiling an explicit acceptance step before merge. If it janks, prefer hiding .arc-label/.ring-label during the active gesture (classList toggle on .sunburst, restore on settle) over CSS-transform blur — momentarily absent labels offend the owner's legibility taste far less than blurry ones.
- **[minor]** The zoom-reset chip never displays as specced: the base rule .zoom-reset { display:none } is not overridden in the ≤720px block (the listed phone styles omit a display property), so the chip is invisible on the only platform that has zoom.
  - Fix: Add display:inline-flex to the ≤720px rule. Keep .zoom-reset[hidden]{display:none} — its 0,2,0 specificity beats the phone rule regardless of cascade order, so hidden-toggling still works.
- **[minor]** Chip placement collides with bottom chrome: fixed bottom-left 14px sits on top of the in-flow legend + status rows (app.css:93, 137); the legend only reserves padding-right:140px for the FAB (app.css:292), nothing on the left. The owner is exacting about this kind of overlap and it is screenshot-visible.
  - Fix: Raise the chip (bottom: calc(52px + env(safe-area-inset-bottom))) or stack it above the panel-fab on the right; verify with iPhone-15 WebKit screenshots at k=1 and k>1 in both themes.
- **[minor]** Reset-grammar inconsistency: the hub/gap tap (data-action="reset", render.ts:171 → app.ts:375-376) is the established nuclear reset (focus+match+pin), but when focusId is already undefined it leaves a k>1 zoom in place — the geometry-key rule only resets view when focusId/slam actually changes. A user who pinched without focusing taps the hub expecting everything to snap back and it doesn't.
  - Fix: Also set view=IDENTITY (and call applyView) inside the reset branch at app.ts:375-376. This keeps the ephemeral-view model while making the one existing 'reset everything' affordance honest.
- **[minor]** ESC ladder (app.ts:421-428) never clears geometric zoom. Defensible since the feature is phone-targeted, but iPad-with-hardware-keyboard exists and the user's original complaint is specifically about unintuitive back/ESC behavior; an invisible layer ESC can't pop worsens that.
  - Fix: Either insert zoom-reset as one ESC layer (after pinnedId, before focusId — one press, one layer) or document the deliberate omission in the openQuestions answer; don't leave it unaddressed.
- **[minor]** Mid-gesture draw() (live fetch resolving, app.ts:235/241 — the design cites 235/239) removes the implicitly-capturing arc element in the innerHTML swap; WebKit then fires pointercancel/lostpointercapture for the touch pointers, aborting the pinch entirely, not just snapping the visual. The design treats this as a cosmetic snap.
  - Fix: Accept the pointercancel path (the Map cleanup already handles it) but add a one-boolean deferral: skip draw() while pointers.size === 2 and flush the pending draw on gesture end. ~5 lines, removes the failure mode during live tournaments.
- **[minor]** Implementation plan references "mount()" — no such function exists; the entry point is createApp(root) at app.ts:47, and gesture state must live next to lastTouchTs (app.ts:267) inside it so the AbortController signal idiom (app.ts:51-52) covers the new listeners.
  - Fix: s/mount()/createApp()/ in the plan; declare view/pointers/lastGestureTs alongside lastTouchTs and pass { signal } on every new listener as already promised.
- **[minor]** Crispness verification as described won't prove anything: a default Playwright screenshot at DPR 1 cannot distinguish vector-crisp from rasterized text at k=2.5; the legibility claim is the owner's core acceptance criterion.
  - Fix: Use the iPhone 15 device descriptor's deviceScaleFactor (3) for screenshots and crop/zoom an outer-ring label region; synthetic PointerEvent dispatch via page.evaluate will drive the delegated handlers fine since nothing in app.ts checks isTrusted.

CLOSE: ✕ chip, ESC, tap elsewhere. FOCUS: the ⊕ chip, plus the swollen arc is itself the natural tap-to-zoom target (dovetails with problem 2).

**Pros:** The chart IS the UI — nothing appears over or beside it, on any viewport, one grammar everywhere; Would be the most distinctive, ownable interaction in the app if executed well

### Critique verdict

The recommended design (Alternative A, unified context strip + on-demand detail tier) fundamentally holds: every load-bearing feasibility claim I checked against the real code is true (stacking insight in .side, stale "replaces" comment, ghost focus button, z-ladder, ESC ladder, ≤960px readout dock, reset semantics), it fits the draw()/data-action/full-innerHTML idiom, and it avoids the WebKit-textPath, fixed-px-overlay, and touch-hover bug classes the owner has been burned by. However, the implementation plan as written has two major defects that would ship visible bugs — the readout's `has-match` hiding class is destroyed by updateReadout's outerHTML swap (app.ts:100-110), and the desktop `.ro-float` corner card (absolute in .sunburst, app.css:160,191) will sit on top of the new strip while the 721–960px band shows both strips — plus several minor wiring gaps (wrong file for AppState, incomplete detailExpanded clearing, no resize-safe surname switching). All are fixable within the design; nothing requires choosing a different alternative.

- **[major]** The plan hides the mobile docked readout by adding a 'has-match' class at draw() time, but updateReadout (app.ts:100-110) replaces the element via `el.outerHTML = renderReadout(info, roCls(idle))` and roCls (app.ts:97) knows nothing about the match state. On a phone, the first finger-drag across the bracket (pointermove fires on touch because the svg is touch-action:none, app.css:70) resolves a new occupant, swaps the readout, drops 'has-match', and the supposedly-hidden 40px readout strip pops back in above/below the match strip — a mid-gesture layout jump. The design lists this as a vague risk; it is a certain bug as specified.
  - Fix: Make the class part of the readout's derived state, not a draw()-time injection: add `isMatch` to the `ctx` object (app.ts:63, set at app.ts:201) and fold it into roCls, e.g. `const roCls = (idle) => "ro-float" + (idle ? " ro-idle" : "") + (ctx?.isMatch ? " has-match" : "")`, so every outerHTML rewrite re-emits it. Add a unit test that pointermoves an arc while a match is selected and asserts the readout still carries has-match.
- **[major]** Strip placement collides with the existing readout geometry in two viewport bands the plan doesn't handle. (1) Desktop >960px: `.ro-float` is absolutely positioned at top:10px/left:10px relative to `.sunburst` (position:relative at app.css:160, rule at app.css:191), and the readout markup is a SIBLING of `.chart` inside `.sunburst` (app.ts:215-216) — injecting the strip as the first child of `.sunburst` puts the hover card directly on top of the strip's left half whenever a match is selected. The plan's claim 'desktop ro-float corner card keeps working for hover' is false as specified. (2) 721–960px: the readout docks as a static order:-1 strip from 960px down (app.css:196-200), but the plan's `.readout.has-match { display:none }` lives only in the ≤720px block — so tablets/narrow windows stack BOTH strips (~130px of chrome) above the chart.
  - Fix: (a) Move the hide rule into the ≤960px media block so the dock band never double-stacks. (b) For >960px, have draw() add a modifier class on `.sunburst` (e.g. `has-strip`) and either offset the corner card below the strip (`.sunburst.has-strip .ro-float { top: calc(var(--strip-h, 64px) + 16px); }` — viable only if the desktop detail tier renders BELOW the chart or as its own block so strip height stays near-constant) or, simpler and defensible, hide the corner card while a match is selected since the strip already names both players and hover still lights paths. Whichever is chosen, screenshot-verify at 1280×800 and ~900px width — the owner checks these.
- **[minor]** Grammar collision with the empty-chart unpin branch: app.ts:284-289 releases pinnedId on any click whose target has no [data-action] and is inside `.sunburst` (`t.closest(".sunburst")`). The strip is injected inside `.sunburst`, so tapping dead space in the strip (score text, player names, padding) silently unpins the player's lit path while the strip stays open — surprising residue, and on touch it also re-arms the pin-first tap grammar for the same player.
  - Fix: Scope the branch to the wheel: change the guard to `t.closest(".chart")` (the readout and centerId are pointer-events:none, so .chart taps are genuinely chart taps), or add `&& !t.closest(".match-strip")`. Add a test: click strip dead-space → pinnedId unchanged.
- **[minor]** Implementation step 2 says 'src/state.ts — add detailExpanded?: boolean to AppState', but AppState is defined in src/app.ts:19-36, not state.ts. Mechanical, but an executor following the plan literally will hunt in the wrong file (state.ts exports SunNode/Snapshot helpers only).
  - Fix: Add `detailExpanded: boolean` to the AppState interface at app.ts:19-36 and initialise it false in the state literal at app.ts:55-59.
- **[minor]** detailExpanded clearing is incomplete. The plan clears it in close-detail (app.ts:371), resetSelection (app.ts:246-249), the inspect match-switch (app.ts:362), and a new ESC rung — but misses two branches that also clear selectedMatchId: the colordim switch (app.ts:331-332) and the reset branch (app.ts:375-376). A stale detailExpanded=true there means the NEXT match a user opens starts pre-expanded (full sheet over the chart on mobile), defeating the strip-first grammar.
  - Fix: Route every match dismissal through one helper (`const closeMatch = () => { state.selectedMatchId = state.selectedNodeId = undefined; state.detailExpanded = false; }`) and call it from close-detail, colordim, reset, ESC, and resetSelection. Alternatively clear detailExpanded inside the inspect branch unconditionally when the match id changes.
- **[minor]** The plan specifies 'full names ≥960px, surname() below' and 'marathon scores at 393px → surname()', implying viewport-conditional rendering — but there is no resize handler anywhere (CSS does all scaling; confirmed: no resize/matchMedia listeners in src/app.ts), so a JS-side surname decision is stale after any window resize and off-pattern to add.
  - Fix: Render both forms and let CSS pick: `<span class="ms-name"><span class="nm-full">Carlos Alcaraz</span><span class="nm-short">Alcaraz</span></span>` with `.nm-short{display:none}` and the inverse inside the ≤960px (or ≤720px) block. Zero new JS, resize-safe, and screenshot-verifiable in both states.

On an iPhone 15 the chart loses ~40-50px of height, and the wheel is already the scarce resource the owner protects.
  - Fix: Collapse to two rows on ≤720px: fold 'Details ▾' into the caption row (caption left, Details ▾ / ⊕ Zoom / ✕ right), keep the matchup row single-line with the dual-span surnames. Verify with the planned iPhone 15 WebKit screenshots that R128 labels remain at the current scale or document the accepted loss.

The owner verifies mobile in WebKit screenshots, but tap ergonomics won't show in a screenshot.

- **[minor]** Test debt is real but slightly understated: besides render-detail.test.ts:33-34 (asserts data-action="focus" and "close-detail" — verified), app.test.ts has four .match-insight assertions including line 289 which explicitly asserts `.side .match-insight` exists ('keeps the lens panel when a match opens (insight stacks below it)') — that test's PREMISE is deleted by the redesign, not just its selector. smoke.test.ts has no match-insight references (verified by grep), so the plan's 'grep smoke.test.ts' is harmless but unnecessary.
